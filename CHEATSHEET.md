# Git-cheatsheet для вайбкодера

## Три золотых правила

1. **`main` — священная корова.** AI на ней не работает. Эксперимент = ветка.
2. **Одна ветка — либо твоя, либо AI.** Не редактируй вручную ту ветку, где сейчас работает агент.
3. **Diff обязателен на критичном.** На тривиальном — можно доверять, но проверь логику AI хотя бы глазами.

## Ритуал перед AI-задачей

```bash
git switch main          # вернулись на main
git pull                 # подтянули свежее
git switch -c feature/x  # новая ветка под задачу
# → запускаем AI, читаем diff, коммитим осмысленные куски
```

## Команды, которые тебе понадобятся каждый день

### Смотрим состояние
| Команда | Что делает |
|---|---|
| `git status` | что изменилось / что в стейдже |
| `git diff` | что наменял с последнего коммита (unstaged) |
| `git diff --staged` | что собираюсь закоммитить |
| `git diff <A>..<B>` | разница между двумя коммитами/ветками |
| `git log --oneline` | компактная история коммитов |
| `git log --oneline --graph --all` | визуальный граф со всеми ветками |

### Коммитим
| Команда | Что делает |
|---|---|
| `git add <file>` | положить в стейдж |
| `git add -p` | по кускам, интерактивно (читаешь diff и решаешь) |
| `git commit -m "msg"` | закоммитить стейдж |
| `git commit -am "msg"` | add + commit за один шаг (только tracked files) |

### Откатываемся
| Команда | Что делает |
|---|---|
| `git restore <file>` | откатить unstaged изменения файла |
| `git restore --staged <file>` | убрать файл из стейджа |
| `git reset --hard HEAD~1` | откатить последний коммит (теряем работу!) |
| `git revert <commit>` | сделать новый коммит, отменяющий старый (безопасно для общих веток) |

### Ветки
| Команда | Что делает |
|---|---|
| `git switch -c feature/x` | создать и переключиться |
| `git switch main` | переключиться |
| `git branch` | список локальных веток |
| `git branch -d feature/x` | удалить ветку (после мержа) |
| `git merge feature/x` | влить ветку в текущую |

### Удалёнка
| Команда | Что делает |
|---|---|
| `git push -u origin feature/x` | первый push новой ветки |
| `git push` | дальнейшие push'и |
| `git pull` | подтянуть свежее с remote |
| `git fetch` | подтянуть, но не мержить |

## PR-флоу (через GitHub UI)

1. `git push -u origin feature/x` → в браузере появится плашка "Compare & pull request"
2. Открываешь PR, описываешь что и зачем
3. Сам или ревьюер смотрит вкладку **Files changed**
4. Кнопка **Squash and merge** (предпочтительнее обычного merge: один PR = один коммит в main)
5. **Delete branch** прямо в UI
6. Локально: `git switch main && git pull && git branch -d feature/x`

## Worktrees (параллельные AI-агенты)

```bash
git worktree add ../vibe-todo-feat-auth feature/auth
# теперь у тебя ДВЕ рабочие папки, в каждой свой Claude Code, своя ветка
```

Удалить:
```bash
git worktree remove ../vibe-todo-feat-auth
```

## Если случилась беда

| Ситуация | Что делать |
|---|---|
| Я наменял в файле — хочу всё вернуть | `git restore <file>` |
| Я закоммитил мусор — последний коммит | `git reset --hard HEAD~1` (если не запушено) |
| Я хочу отменить коммит из истории | `git revert <commit>` (создаёт компенсирующий коммит) |
| Я не знаю что я сейчас вижу | `git status` |
| Я залез не туда и хочу выйти | `git merge --abort` / `git rebase --abort` |
| Я что-то потерял | проси Claude Code посмотреть `git reflog` |

## Бонус: git через Claude Code

Всё что выше — можно **попросить** агента в чате:
- «откати последние 2 коммита»
- «найди коммит, где я добавлял авторизацию»
- «помоги разрешить этот конфликт»
- «открой PR с описанием на основе моих коммитов» (через `gh`)

Цель этой шпаргалки — не заучивать команды, а понимать что просить и проверять что агент сделал.
