Начало

Уже знаем

Мы уже умеем clone, add, commit, push и что-то слышали про ветки. Этого хватит, чтобы код где-то лежал — но не хватит, чтобы спокойно работать с AI.

Зачем лекция

Без дисциплины и контроля версий проект превращается в кашу после одного неудачного промпта.

Сегодня учимся поддерживать порядок: работать с веткми, читать diff'ы, пускать параллельных агентов через worktrees и что делать, когда всё-таки получили конфликт.

Главный takeaway

Три золотых правила

1

Коммить часто, мелко, откатываемо

Каждый коммит — точка отката. Агент ушёл не туда → reset, а не разбор завалов.

2

main — священная корова

AI на ней не работает. Никогда. Только через ветки.

3

Одна задача за раз

Не «сделай приложение», а «добавь кнопку логина». Узкий шаг — понятный результат.

Если забудем всё остальное — помним эти три.

Diff живёт в трёх местах

Diff — главный инструмент проверки AI. Смотрим перед каждым коммитом: это момент, где видно, что агент на самом деле наменял.

git diff A..B что изменилось между двумя точками — «что AI сделал за час»

Откат

Убили? Загрузи последний сейв. Главное — чтобы сейв был.

Откат

Где мы находимся — такой и откат

Уровень Когда Команда
Не закоммичено поменяли, но пока не коммитили git restore <file>
Закоммичено локально commit был, push — НЕ был git reset --hard HEAD~1
Уже на GitHub запушено, другие могли увидеть git revert <commit>

⚠️ reset --hard выкидывает работу. Используем, только когда уверены.

Ветки

Зачем

Без веток = всё на main. Когда AI ломает main = «оно что-то нажало — и все исчезло!»

С ветками:

  • main всегда работает
  • Каждая задача = новая ветка
  • Готово → PR → review → merge → удалить ветку

Жизненный цикл ветки — часы, максимум день.

Полный цикл

Feature workflow

git switch maingit pull                          # подтянули свежееgit switch -c feature/add-search  # новая ветка# ...работаем, коммитим...git push -u origin feature/add-search# → открываем PR в браузере → ревью# → Squash and merge кнопкой → Delete branchgit switch maingit pullgit branch -d feature/add-search  # локально удалили
Pull Request

PR — это наш Code Review

  • Title — что сделано в одной строке
  • Description — зачем и как тестировать
  • Files changed — собственно review
  • Inline comments — обсуждение конкретных строк
  • Squash and merge — один PR = один коммит в main

Даже в одиночку: PR заставляет посмотреть на свои изменения глазами ревьюера перед мержем.

Live demo · на экране

Ветка → PR → merge

  1. Ветка feature/lecture-demo уже с AI-работой
  2. Push → плашка в GitHub → PR
  3. Title + description → Create
  4. Files changed → разбираем review-tools
  5. Inline comment → Resolve
  6. Squash and merge → Delete branch
  7. Локально: switch main, pull — ветка-призрак удаляется

04

Worktrees

Параллельные AI-агенты на одном репозитории — без драки за файлы.

Проблема

Нужен второй агент на параллельную задачу?

Один репо = одна папка. В этой папке один Claude Code что-то делает. А задач — несколько, и хочется гнать их параллельно.

Без worktrees:

  • Либо второй агент топчется по тому же коду
  • Либо копируем репо в другую папку — и теряем общую историю
Решение

git worktree

git worktree add ../vibe-todo-feat-auth feature/authgit worktree add ../vibe-todo-bugfix-x  bugfix/x
/vibe-todo/

основная папка (main)

feat-auth

Claude Code #1 работает над auth

bugfix-x

Claude Code #2 фиксит баг

итог

Одна история, разные рабочие копии, разные ветки.

Из практики

Рабочий workflow

  1. Research фичи руками или с Claude Code в основной папке
  2. Разбиваем на GitHub issues (gh issue create) — вот оно, правило №3: один issue = одна задача
  3. На каждый issue: git worktree add ../vibe-todo-issue-42 fix-issue-42 → claude
  4. Параллельно идут 3–5 worktrees
  5. Закончили → push → PR → review → merge → удалить worktree

Конфликты

Почти все конфликты — от плохой дисциплины. Поэтому сначала — профилактика.

10 минут
Профилактика

5 правил, чтобы не конфликтовать

1

AI на main — никогда

Не запускаем агента на main. Точка.

2

Ветка = задача = автор

Одна ветка — мы или агент, не оба.

3

Старт всегда одинаков

switch main → pull → switch -c new.

4

Ветки живут недолго

Час, день — не неделя. Дольше = дальше разойдутся.

5

Worktrees для параллели

Два AI не дерутся за одни файлы.

Если всё-таки случилось

Разрешаем конфликт

<<<<<<< HEADtitle: improved version=======title: best in the world>>>>>>> feature/title-b

Merge editor: Accept Current / Incoming / Both.

эвакуация

git merge --abort — всё откатывается.

Claude

«разреши конфликт в этом файле».
